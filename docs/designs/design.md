## Problem Statement

Build a personal web app for Jake's mom — a deep family historian with thousands of documented ancestors going back 1000 years (to John Adams, Lady Godiva) — that lets her *experience* her ancestry interactively in ways static books and Ancestry.com viewers cannot. Specifically: scrub time and watch the bloodline migrate across the world map, get a quiet daily moment with one ancestor, and ask the family record about historical events to see who was alive, where, and what they were doing.

The thing the app gives her that her existing books and tree software do not: time as a controllable axis, geography as a visual layer, and her family's data fused with historical context into narrative she can share.

## What Makes This Cool

The data foundation is unusually rich — most genealogy software is built for people slowly adding ancestors one at a time. Mom already has the tree. That means the app gets to be *about exploration and narrative*, not data entry. With thousands of ancestors across a millennium, a time-scrubber map becomes a moving picture of the bloodline migrating through Europe and into America — visceral in a way no book page can be.

The "wow she texts to her sisters" moment: dragging the year slider from 1620 to 1850 and watching dots bloom across New England as the family arrives, settles, spreads west.

## Constraints

- **Audience:** mom only at v1, on her iPad/laptop. Open-source release possible later — design must not bake mom-specific assumptions into the data layer.
- **Tech reuse:** lean on Jake's existing Next.js + Vercel + Anthropic API muscle (from Venue Hub, Intelligence Brief).
- **Books not available yet:** only GEDCOM at v1. Books-as-oracle is a Phase 2 feature, blocked on scanning the printed books (decent-quality printed books, will need OCR pass).
- **Privacy:** mom's GEDCOM is genealogically sensitive (living relatives' details). Private URL only, no public indexing.
- **Multi-session build:** Jake will implement this across multiple weekend sessions, not in one push.

## Premises (all agreed)

1. **Data sufficiency.** The GEDCOM has enough birth/death dates AND location data to make the time-scrubber visually rich at most eras. (Risk to validate: pre-1700 location data is often sparse. Step zero of implementation is a data audit — count records with valid dates+locations per century.)
2. **App over book.** Mom prefers an interactive app to her existing books for the things books cannot do (time scrubbing, queryable map, daily delight, easy share). Books remain the authoritative narrative source.
3. **Trustworthy AI.** LLM features must ground every claim in GEDCOM (and later, books) data with citations. Confabulation is unacceptable for a serious historian — better to say "I don't have that record" than to invent a detail.
4. **Distribution is trivial.** Private URL on Vercel, mom bookmarks it on her iPad. No accounts, no app store. Data layer stays separate from app code so any GEDCOM can power it later.
5. **App is the right deliverable.** Not a printed coffee-table-book, not a chat-only personal AI, not a weekly email. The interactive map and time-scrubber require an app.

## Approaches Considered

### Approach A: Static-rendered web app — Next.js + Vercel + flat data (CHOSEN)
Next.js, GEDCOM parsed at build time into JSON committed alongside code, Leaflet map, server-side LLM via API routes. All four features fit. Free hosting. Same skills as existing projects. Open-source path stays clean (data layer is replaceable).

### Approach B: Local-first Mac/iPad native app — SwiftUI
Native craft object. Beautiful pinch-zoom on iPad, completely offline-private. Rejected: ~3x effort in unfamiliar stack, fights the open-source-later goal, mom can't share a link with her sisters, RAG over PDFs is harder native.

### Approach C: Personal-AI first, app layered after — wedge play
Phase 0: Claude Project ingesting GEDCOM (and books later). Mom plays with chat in days while web app gets built in parallel. Recommended by office-hours, *not* chosen — Jake wants to build the visual experience he originally envisioned, not the wedge.

## Recommended Approach

**Approach A: Next.js + Vercel web app.** Built across multiple weekend sessions. Each weekend ships one of the four features behind a private URL mom already has bookmarked. The shape:

**Data layer (built once, reused everywhere):**
- A `gedcom-loader` module that parses mom's `.ged` file into a normalized JSON entity store:
  ```
  Person {
    id, name,
    birth: { date, place: { lat, lng, label } } | null,
    death: { date, place } | null,
    events: [{ type: 'RESI' | 'MARR' | 'BURI' | ..., date, place }],  // ordered chronologically
    parents: [personId],   // biological parents, primary first
    children: [personId],  // birth order preserved
    spouses: [personId],
    isLiving: boolean,     // computed by privacy filter (see below)
    sources: [...]         // raw GEDCOM source tags for provenance
  }
  ```
- **GEDCOM version support:** assume GEDCOM 5.5.1 (the common export from Ancestry.com, FamilySearch, RootsMagic). Detect version via `HEAD.GEDC.VERS` at parse time. If mom's export is 7.0 (structurally different: URI-based tags, no ANSEL, different date structures), `read-gedcom` may not parse it cleanly — if so, either (a) ask mom to export as 5.5.1 from her genealogy software (every major tool supports 5.5.1 export), or (b) convert via Gedcom.org's 7.0→5.5.1 converter as a one-time preprocessing step. Do not spend time patching the parser. UTF-8 is the expected encoding; fall back to ANSEL only if `HEAD.CHAR` declares it (5.5.1 only — 7.0 is UTF-8 always).
- **Parser choice:** use `read-gedcom` on npm — it produces typed records and handles most of the schema correctly. `parse-gedcom` alone gives you a raw tag tree, which leaves the normalization work to you. If `read-gedcom` doesn't cover an edge case, layer a small normalization module on top of the raw tag tree.
- **Payload-size target:** keep `people.json` under ~2MB gzipped on the wire to the iPad. With "thousands of ancestors" the full store should fit, but if it balloons past target, split into per-century shards loaded on demand by the time-scrubber, and serve only the entity ID + name + dates on first paint (full record loaded on dot-click).
- **Geocoding pass at build time, run locally, not in CI:** every unique `place` string → lat/lng. Nominatim (OSM) is free but capped at 1 req/sec and forbids bulk use. Mapbox and Google restrict persistent caching of geocoded results in their ToS — wrong choice when you want a committed `places.json` cache. **Use LocationIQ or OpenCage** for the bulk pass — the advantage isn't throughput (they cap at ~1–2 req/sec too) but that their ToS explicitly permits persistent caching. Thousands of unique places at that rate is a ~30–60 minute one-time local run. Alternative: self-hosted Nominatim via Docker for a one-time run then shut it down.
- **Where the pass runs:** locally on Jake's machine, output committed to the repo as `public/data/places.json`. Vercel's CI build only reads the cache — it never geocodes — so CI builds are deterministic and fast. New places after a GEDCOM refresh trigger a manual local re-run (delta-only: only geocode places not already in cache).
- **Failure handling:** the geocoder script retries transient errors with exponential backoff up to 3 attempts per place; places that fail permanently are written to `places.json` with `{ lat: null, lng: null, error: "<reason>" }` and the loader excludes them from the map (same as places with no data). The script logs a summary at the end so Jake can hand-fix ambiguous places.
- **Living-person privacy filter** (default-privacy-on):
  - A person is `isLiving: false` only if the GEDCOM has an explicit death event (`DEAT` with date) OR an explicit `_PRIV` / `RESN` tag marking them as non-living
  - Everyone else is `isLiving: true` and gets redacted in the built JSON (name → "[Living]", precise place → country-level, exact dates → year-only or null)
  - This is deliberately aggressive — better to over-redact a dead-but-undocumented great-aunt than to plot a living cousin on a public map
  - Mom can review the redaction list and manually override specific people via a `public/data/overrides.json` file that the loader reads. Schema:
    ```
    {
      "@I123@": { "status": "deceased", "deathYear": 1987 },   // override: this person is dead, use deathYear for lifespan
      "@I456@": { "status": "living" }                          // override: this person is living, keep redacted
    }
    ```
    `deathYear` is optional — if absent, the era-dependent `lifespanDefault` is used. `status: 'living'` is rarely needed (default-privacy-on already handles unknowns) but useful when a GEDCOM mistakenly includes a death record for someone still alive.
- **`gedcom-version-hash`:** on every build, write a `version: "<sha256 of source .ged>"` field into `people.json`. Used as the cache key for narrative caching (Feature 3) so re-runs after a GEDCOM refresh invalidate cleanly.
- Output: `public/data/people.json` (the entity store) and `public/data/places.json` (geocoded place cache). Re-run on GEDCOM update.
- Step zero of implementation: a data audit script that reports per-century coverage of (a) people, (b) people with valid dates, (c) people with geocoded places. This validates Premise 1 before any UI work.

**Feature 1 — Time-scrubber map (v1, weekend 1):**
- Leaflet map covering Europe + North America (the bloodline geography)
- Year slider, range determined by data audit: default to the well-covered range (likely ~1500–2026 for a well-documented American family with European roots). Pre-1500 ancestors are shown but the slider's default-handle position and tick density emphasize the dense-data centuries; full range remains scrubbable.
- For each year *Y*, render a dot for every person where `birth.date.year ≤ Y ≤ death.date.year`. If the person has no death event but has been marked `isLiving: false` by a `_PRIV`/`RESN` tag or `overrides.json` (i.e. they're confirmed dead but the death year isn't in the GEDCOM), use `birth.year + lifespanDefault`; era-dependent: 50 for pre-1700, 65 for 1700–1900, 80 for post-1900. Living-by-default people (no death, not overridden) are redacted by the privacy filter and never reach this renderer. Position by location precedence:
  1. Most recent `RESI` (residence) event before *Y*
  2. Else most recent marriage place before *Y*
  3. Else birth place if *Y* ≥ birth.year
  4. Else death place
  5. Else: don't render (no usable location)
- **Performance pattern:** pre-compute per-decade dot sets at build time (a `dotsByDecade` map: `1840s → [{ id, lat, lng, opacity }]`). Slider tick → look up the bucket → render. This avoids per-tick recomputation across thousands of records on iPad Safari.
- **Renderer choice:** start with Leaflet + `Leaflet.markercluster` for zoomed-out clustering. If iPad jank appears at thousands of dots, switch the active layer to `deck.gl`'s ScatterplotLayer over Leaflet (actively maintained, handles tens of thousands of points on mobile). Decide after the v1 data audit reveals actual record counts. (Avoid `Leaflet.glify` — unmaintained, known issues with recent Leaflet versions.)
- Click a dot → side panel (the person-card), see spec below
- **Edge cases to handle, not skip:** people with birth date but no place (don't render until they appear somewhere via the precedence rules above); people whose place changes (handled by precedence); people with date ranges instead of exact dates (render for the whole range with `opacity: 0.4`, normal opacity for exact dates); the "John Smith of Kent" duplicate-name problem (key on GEDCOM ID, never on name)

**Person-card (side panel, used by Features 1, 2, 3):**
- Header: full name, birth–death years (`1766 – 1841`), optional relationship-to-mom line (`your 5x-great-grandmother`, computed from the graph)
- Life line: chronological event list (birth, residences, marriages, death) pulled from `events[]`, each with place + date
- Family: parent(s) clickable to load their card, children list clickable, spouse(s) clickable
- Navigation: up-arrow (primary parent — first in `parents[]`, per GEDCOM convention the biological father if set, else mother), down-arrow (firstborn child — first in `children[]`), left/right (siblings in birth order). Keyboard: arrows or `h/j/k/l`. Breadcrumb trail at top of side panel for back-navigation.
- Share: a "copy screenshot" button that renders the current card + map-position via `html-to-image` to PNG, copies to clipboard for pasting into Messages/email
- Empty state: if a person has no location data, the card still opens but the map doesn't pan to a dot — card header notes "location unknown"

**Feature 2 — On-this-day (weekend 2):**
- Pure server-side computation: given today's MM-DD, find all people whose birth/death/marriage falls on this day across history
- Pick one and rotate through across the year. **Rotation store: Upstash Redis via Vercel Marketplace integration.** (Vercel functions cannot write to the filesystem / git — any "committed state" approach is incompatible with serverless runtime.) One Redis key: `rotation-state` holding JSON `{ "year": 2026, "seenIds": ["@I123@", "@I456@"] }`. Env vars set by the integration: `KV_REST_API_URL`, `KV_REST_API_TOKEN`. Free tier (10k commands/day) is 100x more than needed for a single-user daily read-modify-write. The daily API route does: read → if `year !== currentYear` reset to `{year: currentYear, seenIds: []}` → pick an unseen eligible ancestor → append to seenIds → write. Per-device localStorage is wrong here — mom uses iPad and laptop, both should agree.
- **Tiebreaker:** when multiple eligible ancestors share today's date, pick deterministically by GEDCOM ID (lexicographic) from the unseen set. When `seenIds` includes every eligible person for today's date, reset the today-subset (but keep the annual rotation intact).
- Render a beautifully typeset card grounded *only* in GEDCOM data: "On April 15, your 5x-great-grandmother Sarah Adams (b. 1766, d. 1841) was 23 today, living in Salem, MA." No LLM-generated world-historical context in v1 — Premise 3 (citation required for every claim) means free-form historical context is out of scope until there's a grounded source for it. (See "Maybe later" section below.)

**Feature 3 — Event narrative (weekend 3):**
- Curated list of historical events stored at `public/data/events.json`, hand-authored. Schema:
  ```
  Event {
    id,                    // "american-civil-war"
    title,                 // "American Civil War"
    dateRange: { start, end },  // ISO years
    geoBbox: { minLat, minLng, maxLat, maxLng },  // bounding box for proximity filter
    scopeTag: 'regional' | 'national' | 'global',
    blurb                  // one-sentence human context
  }
  ```
  Seed list: American Revolution, Civil War, WWI, WWII, Great Depression, Plague of London, Great Famine (Ireland), French Revolution. Add as mom requests.
- For a selected event: query the entity store for all ancestors alive during the date range, filtered by `geoBbox` overlap with their known location at event start.
- **Grounding pattern:** pass Claude a structured JSON list of eligible ancestors — each with `id`, `name`, `ageAtEventStart`, `locationAtEventStart`, `occupationIfKnown`, `deathDuringEvent` (boolean) — and the event's structured record. The prompt explicitly constrains Claude to narrate *only* facts present in the structured record; for each factual claim, require Claude to emit an inline citation token matching `[person:@I\d+@:(age|location|occupation|deathYear|name)]`. Example: `Sarah Adams [person:@I123@:name] was 23 [person:@I123@:age] living in Salem [person:@I123@:location].` No imputing occupations, deaths, emotions, or actions not in the structured data.
- **Validation:** after the LLM responds, server-side code regex-parses every citation token, looks up the source field in the structured input, and confirms the claim matches. Any factual sentence (non-trivial clause) without a matching citation triggers a single retry with a stricter "you omitted citations on sentences X and Y — regenerate with citations" message. Second failure → render with a warning banner "narrative generation partially failed, some claims may be unverified" rather than failing silently.
- **Rendering contract:** the client parses citation tokens from the validated output, renders the surface text with token spans hidden, and shows the raw GEDCOM field as a tooltip on hover over the adjacent phrase.
- Render narrative with clickable name → person card. Cache narratives by `(event-id, gedcom-version-hash)` (defined in the data layer section) so re-renders are free and mom's reading is fast.

**Feature 4 — Books-as-oracle (Phase 2, blocked on scanning):**
- Scan the printed books with phone scanner (~1 day per book). **OCR quality is the major risk** — Tesseract on old typefaces, mixed fonts, and tables typically produces noisy output that poisons the RAG index. Plan: scan one sample page, run it through Tesseract AND a paid OCR (Adobe Acrobat OCR, Mathpix, or Azure Document Intelligence), eyeball both outputs. If Tesseract is materially worse, budget for paid OCR (Adobe is the safest commercial bet for printed-book quality at this scale).
- Embed chunks via Voyage AI (Anthropic's recommended embeddings partner — Anthropic does not ship a first-party embeddings API) into a flat-file vector store with cosine search (sufficient for mom-only scale; thousands of chunks is trivial in memory)
- Query flow: user question → retrieve top-N chunks from books + relevant entities from GEDCOM → Claude synthesizes answer with citations to specific book pages and GEDCOM IDs
- Critical correctness pattern: GEDCOM is the authoritative entity index, books are narrative sources. When the LLM mentions a person, it must reference a GEDCOM ID — never invent a person from book text alone.

**Maybe-later (post-v4, only if mom asks):**
- **Daily email push** of the on-this-day card via Resend at 7am her time. Pull (open the app daily) is the v1-v4 default; switch to push only if she asks for it.
- **LLM-generated world-historical context** added to the on-this-day and event-narrative features (e.g. "George Washington had been president for two weeks"). Requires a grounded source for general history (curated dataset or a citation-backed encyclopedia API) to satisfy Premise 3. Out of scope until that source exists.

## Open Questions

- **GEDCOM data audit results.** Until we run the data-audit script, Premise 1 is unverified. If pre-1700 has, say, 10 people with locations vs 500 in 1800–1900, the time-scrubber feels empty in early eras. Mitigation: graceful empty-state messaging, or supplement sparse eras with the historical-context overlay (e.g. show major historical events on the timeline even when no ancestor data is on screen).
- **Geocoding accuracy.** "Wiltshire, England" geocodes to a county centroid which is fine. "Plympton St. Mary, Devon" might fail. Need to manually clean ambiguous places — probably 1-2 hour pass after the first geocode run.
- **LLM prompt craft for event-narrative.** First version will likely feel either generic ("the family lived through difficult times") or dangerously specific (invented details). Plan to iterate the prompt with mom's review of the first few outputs before considering the feature done.
- **Where does the GEDCOM live in the repo?** If open-sourced later, mom's GEDCOM can't be in git. Plan: built artifacts in `public/data/` are not committed (gitignored); the source `.ged` lives outside the repo entirely. Build script reads from a configurable path. (Built JSON contains living-relative records — even with the privacy filter applied, treat as private until mom reviews.)

## Eng Review Additions (2026-04-16)

Architecture-level additions resolved in /plan-eng-review that did not exist in the original design doc:

### Leaflet + Next.js SSR
Every Leaflet-using component must be dynamically imported with `{ ssr: false }` and marked `"use client"`. Leaflet requires `window` and `document`; server-side rendering throws. Pattern:

```
const MapView = dynamic(() => import('@/components/MapView'), { ssr: false });
```

### LLM narrative streaming (v3)
`/api/narrative/[id]` uses streaming response — Anthropic SDK's `stream: true` piped into a Next.js `ReadableStream`. First token returns in <1s regardless of total generation latency. Avoids Vercel Hobby's 10s function timeout (Claude narrative can take 10–30s). Better UX: mom watches the narrative write itself. If the stream is interrupted mid-generation, render what arrived + a small italic "the record grows silent" banner.

### Narrative cache eviction
Cache keyed on `(event-id, gedcom-version-hash)`. On every build, after rebuilding `entities.json`, a cleanup pass purges cached narrative entries whose hash does not match the current build's hash. Deterministic, no TTL needed. One query per cache store at build time.

### RAG chunking defaults (v4)
Stories are chunked at **750 tokens with 100-token overlap**, section headings preserved as chunk metadata when present. Default for narrative content of this length. Embeddings via Voyage AI (Anthropic's recommended partner; Anthropic does not ship a first-party embeddings API).

### Upstash REST client
Rotation-state module uses `@upstash/redis` (REST client), NOT `ioredis`. The REST client is connectionless — no cold-start latency in serverless functions. Environment variables `KV_REST_API_URL` and `KV_REST_API_TOKEN` are set by the Vercel Marketplace Upstash integration.

### Required tests before v1 ships
17 critical paths were identified in /plan-eng-review's coverage diagram. The privacy filter's 8 rules each need a dedicated test against a fixture tree containing: one explicit-living (born 2000, no DEAT), one explicit-deceased (DEAT present), one _PRIV-tagged, one RESN-tagged, one pre-1700 no-DEAT, one MARR-only pre-1900 (no birth year), one override `status: 'living'`, one override `status: 'deceased' + deathYear`. Assert every rule fires in rule-order.

Additional required tests: kinship label correctness (direct lineage, collaterals, disconnected), arc clip-at-Y (no future segments), Leaflet dynamic import renders without `window is not defined`, shared-password middleware redirect.

### Build-time smoke test (critical gap closure)
`scripts/build-data.ts` emits a non-zero exit if the `isLiving: true` redacted count is zero. Mom's tree includes several living relatives; a zero count means the privacy filter is broken. Hard gate before any build succeeds.

### Boot-time Claude API health check
On server boot (or first request), run a 1-token Claude API call. If it 401s, set a flag that surfaces a banner on the narrative and oracle pages: "Narrative features temporarily unavailable — check API key." Fail loudly, not with a 500.

## Design Review Additions (2026-04-16)

Visual direction locked via wireframe at `docs/designs/wireframe.html`. Aesthetic: **weathered archival heirloom**. Cream parchment, deep brown ink, burnished gold, ox-blood accents. Cinzel display caps + Cormorant Garamond body. Gold filigree corner ornaments. Brass slider handle.

Full tokens in `DESIGN.md` and `app/globals.css`.

### Decisions added to the plan

0. **Hybrid interaction model: sidebar primary, popover at dot as spatial anchor.** Dots on the map are unmistakably clickable. On click, two things happen together: (a) the dot gets a strong gold halo and a small cream floating label tags it with `Name — years` (e.g. `Sarah Adams · 1697–1762`), anchored at the dot like a hand-lettered map tag, and (b) the right sidebar updates with the full record (life events, sources, kinship, family navigation). On hover (desktop only), a softer halo and subtle tooltip preview the same label without committing to a selection. The floating tag at the dot is itself clickable — tapping it behaves identically to clicking the dot. This preserves the JUUCE-map instinct (map element is the primary affordance) while keeping the rich content surface of a sidebar for a serious genealogist.
1. **First-visit hint** ("Drag the year to follow the bloodline") shown once, auto-dismisses on first slider drag. Never shown again unless mom explicitly resets.
2. **Map tiles:** Stadia-hosted Stamen Watercolor (free tier for non-commercial, attribution required). Hand-painted feel matches the aesthetic. Switch to Mapbox Custom Style only if open-source release creates scale issues.
3. **Default year on first load:** 1700. Dense era, emotionally accessible for American descendants, enough pre-/post-data for mom to scrub in either direction.
4. **Kinship on dots:** label ONLY on hover, NEVER painted directly on dots. Rendering labels on thousands of points doesn't scale on iPad.
5. **iPad portrait layout:** below 1024px viewport, sidebar slides in from the right on dot-select (not always visible). Swipe-from-right-edge or Escape dismisses.
6. **Dot tap targets:** 24×24px invisible hit wrapper around each visual 9px dot. Apple HIG requires ≥44pt but the visual dot stays small for the archival aesthetic; finger-tap zone is generous.
7. **Gold is decoration only.** `#A68835` on `#F1E6D2` = 3.2:1 — fails WCAG AA for body text, passes for large/decorative. All body copy uses `--brown` or `--deep` on cream.
8. **No component library.** No shadcn, no Material UI, no Chakra, no Radix. Ship plain semantic HTML + project CSS. Every component library's defaults are AI-slop vectors for this aesthetic.

### Interaction state coverage

| State | What mom sees |
|-------|---------------|
| First open, nothing selected | Sidebar eyebrow: "Begin here"; body: "Select an ancestor, or drag the year." |
| Year with zero ancestors | Map stays, cartouche reads year, sidebar: "No ancestors appear in [year]." Never blank. |
| Loading (GEDCOM fetch) | Cream page with engraved title + italic "Gathering the record..." No spinner. |
