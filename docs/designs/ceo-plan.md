## Data Audit Findings (new since design doc)

The design doc was written before the GEDCOM was available. Data audit revealed the tree is substantially richer than the design assumed:

- 3,491 individuals, 1,498 families, 1,828 media files, 887 cited sources
- Birth year coverage dense from 1400 forward (87 in 1400s, 624 in 1500s, 919 in 1600s, 906 in 1700s, 614 in 1800s)
- 92% of individuals have birth places (3,225 / 3,491)

- 375 media files are "stories" (HTML/PDF/docx narratives attached to specific ancestors)
- Only 7 occupation tags across 3,491 people (occupation data effectively absent)
- FILE references in GEDCOM are empty paths, media requires separate Ancestry ZIP export

### Source-citation architecture (v1, foundational)

Two distinct citation surfaces share a common helper but are architecturally separate:

**Static source lookup** (person-cards, every fact):
- Build-time pass indexes all top-level `SOUR` records by ID into `public/data/sources.json`.
- For each indexed SOUR, extract `TITL` (primary label) with `REPO` as fallback when TITL is missing; include `PAGE`/`CITN` if present for detail tooltips.
- Per-fact citations attach at the event level: each `events[]` entry and `birth`/`death` gets a `sources: [{ sourceId?, page?, inlineText?, quay? }]` array covering both GEDCOM citation forms Ancestry produces:
  - **Pointer form** (`SOUR @S123@`): extract `sourceId = "@S123@"`, plus child `PAGE`, `DATA`/`TEXT`, `QUAY` (certainty) if present. Resolved via `sources.json` at render time.
  - **Inline form** (`SOUR` followed by free text, no `@...@` pointer): synthesize a citation with no `sourceId`, `inlineText` = the SOUR's inline text plus any `TEXT`/`PAGE` children. Render as a source snippet without a stable ID.
- **Parser caveat:** `read-gedcom`'s typed output should distinguish pointer vs inline SOUR, but verify against mom's actual export. If the typed output conflates them, drop to the raw tag tree for citation extraction and walk the SOUR children directly.
- Runtime helper: `sourceLabel(citation)` returns `{ title, repository?, page?, quay? }` for pointer-form, or `{ snippet, page? }` for inline-form. Zero LLM involvement. Hover any fact â†’ tooltip.

**LLM-validated citation tokens** (event narrative, v3 only):
- The design doc's `[person:@I\d+@:(age|location|...)]` regex system is unchanged. Post-response validation + retry loop stays in place.
- Both surfaces call `sourceLabel()` when rendering, so the UI tooltip content is consistent.

### Kinship module (v1, shared foundation for expansions #2 and #5)
- Module exports `labelFor(personId): string` and `descentPath(personId): PersonId[]`.
- Both consume `MOM_GEDCOM_ID` (env var, resolved once at build into a module-level constant).
- Label style (resolved, not an open question):
  - **Direct lineage (mom's direct ancestor or direct descendant):**
    - Generation 1: `mother`, `father`
    - Generations 2â€“5: `grandmother`, `great-grandmother`, ..., `5x-great-grandmother`
    - Generations 6+: shorthand `6g-grandmother`, `15g-grandmother`, up to `30g-grandmother`
    - Spouse of a direct ancestor: prefix `wife of your Ng-grandfather`, etc.
  - **Collateral (sibling of a direct ancestor, or their descendants â€” aunts/uncles/cousins):**
    - `Nth cousin Mx removed` (e.g., `4th cousin 3x removed`) for cousin relationships
    - Formula: find the MRCA (most recent common ancestor) of mom and target. Let `gm` = generations from mom to MRCA, `gt` = generations from target to MRCA. Then `N = min(gm, gt) - 1` (1st-cousin means MRCA is 2 generations up for both), and `M = |gm - gt|` (removed steps). If `min(gm, gt) == 1`: target is a sibling of a direct ancestor â†’ use `great-aunt` / `Nx-great-aunt` form, not cousin form.
    - `great-aunt`, `2x-great-aunt`, ..., `Nx-great-aunt`/`-uncle` for direct siblings of an ancestor
  - **Half-relationships, step-relationships, in-laws, and spouses of collaterals:** v1 returns the base label without qualification (a half-sister is labeled `sister`, a step-parent is labeled `parent`). Deferred to a future `labelFor(id, { precise: true })` mode. These edge cases exist in the data but are not material to the emotional core of the v1 experience.
  - **No direct connection found (should never happen in mom's own tree but guard anyway):**
    - Return `"related via your tree"` (non-fatal fallback)
  - **Zero-distance:** `"you"` when `personId === MOM_GEDCOM_ID`
  - Prefixes `"direct ancestor"` / `"collateral relative"` are NOT applied by default. Those are reserved for a future `labelFor(id, { verbose: true })` mode that clarifies branch type in UI contexts where it matters (e.g., line-of-descent viewer header).

### Privacy filter â€” era-based auto-deceased (CRITICAL revision to design doc)

The design doc's default-privacy-on filter required an explicit death event, `_PRIV`/`RESN` tag, or override for `isLiving: false`. With the actual data (1,600+ ancestors born pre-1700 with no DEAT records because they died centuries ago), that rule would redact the majority of the tree and break every feature.

Revised rule (evaluated in this order):
1. **Override wins:** if `overrides.json` has `status: 'living'` for this person â†’ `isLiving: true` (redact). This is the force-living branch for the rare "GEDCOM mistakenly includes a DEAT for someone alive" case from the design doc.
2. **Override wins:** if `overrides.json` has `status: 'deceased'` â†’ `isLiving: false` (render). Use `deathYear` if provided, else `lifespanDefault`.
3. **Explicit DEAT present** â†’ `isLiving: false` (render), use the DEAT year.
4. **Explicit `_PRIV` or `RESN` tag** â†’ `isLiving: true` (redact).
5. **Era rule (birth year):** `birth.year < (currentYear - 120)` â†’ `isLiving: false` (render), use `lifespanDefault` (era-dependent: 50 pre-1700, 65 for 1700â€“1900, 80 post-1900) for rendering bounds.
6. **Era rule (any event year) â€” fallback for people with no birth year:** if any event on this person (MARR/DEAT/RESI/BURI) has `year < currentYear - 120` â†’ `isLiving: false` (render). Anchor `lifespanDefault` to the earliest known event year. This saves pre-1900 collateral relatives who are documented only by marriage records, not birth records.
7. **Default:** `isLiving: true` (redact).

**Auto-deceased review:** the build script emits `public/data/auto-deceased-report.json` listing every person flipped to deceased by rules 5 or 6, with their name, the triggering event, and their GEDCOM ID. Mom (or Jake) scans the report after the first build and adds entries to `overrides.json` for any false positives (e.g., a recently-born descendant whose birth year is missing for some reason). This preserves the design doc's explicit-review-workflow principle without blanket-redacting all the pre-1700 ancestors.

This order ensures overrides always beat heuristics, explicit GEDCOM signals beat era defaults, and the era rules save ancient ancestors (with or without birth years) from redaction without overriding mom's own tagged living relatives.

### New feature: v1.5 â€” Migration paths (weekend 1.5)
- Computed per-person trajectory from RESI events ordered by date.
- Rendered as animated arcs on the map, opacity increasing with recency relative to slider year.
- Toggleable in UI (on by default; off mode is just dots for clarity when zoomed in).
- **Data structure (revision):** arcs are NOT the same layer as dots. Precompute `arcsByPerson: { personId â†’ [{year, lat, lng}] }` (ordered location sequence). Dots remain their own per-decade bucket. Two parallel precomputations, two parallel render layers.
- **Render semantics at slider tick *Y*:** the dot layer shows each person at the most recent RESI â‰¤ *Y* (matches the design doc's location precedence rule for dots). The arc layer renders a polyline through the subset of that person's RESI points where `year â‰¤ Y`, ending at the most-recent-â‰¤-*Y* point. **No interpolation past *Y*.** The person is never rendered at a position that depends on a future RESI event. When *Y* is before any RESI for a person, both layers skip rendering them.
- **Per-segment opacity:** each arc segment between two RESI points gets its own opacity value = `clamp(1 - |Y - segmentMidYear| / 50, 0.1, 1.0)`. A segment whose mid-year is within 5 years of *Y* renders at nearly full opacity; one 50+ years away fades to the 0.1 floor. Per-segment (not per-arc) so the "fresh vs. ancient" feel is visible on every trajectory, including long migrations where one person's arc spans 50+ years.

### New feature: v2.5 â€” Line of descent (after weekend 2)
- New route `/descent/:ancestorId` and entry point from any person-card ("show descent from mom").
- Renders the chain mom â†’ ... â†’ ancestor as a vertical ladder. Each row: name, dates, kinship label. **No photo slot in v2.5** (media pipeline not shipped yet; adding a placeholder here would re-introduce the YAGNI scaffolding v1 dropped). Photos added to ladder rows in v4 when the media pipeline lands.
- Reuses the kinship module's `descentPath(id)` (no duplicate BFS logic).
- **Multiple paths (endogamy):** `descentPath()` returns the shortest path by generation count. During the BFS, count how many distinct parent-arrivals land at the target node (O(V+E), cheap). If count > 1, render a simple annotation: `"also reachable via other paths"` (no count number, no enumeration). Do NOT attempt to enumerate all paths â€” that's exponential in a 30-generation graph with endogamy. If mom ever asks to see the other paths, that becomes a v5 feature with a bounded-enumeration approach (cap K=5, depth-limit).
